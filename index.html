How to Draw a Figure on Canvas

CanvasRenderingContext2D.fillRect()

현재 fillStyle에 따라 채워진 rectangle(직사각형)을 그린다. 이 method는 현재의 경로를 수정하지 않고 canvas에 직접적으로 그린다.

CanvasRenderingContext2D.clearRect()

CanvasRenderingContext2D.clearRect()

CanvasRenderingContext2D.strokeRect()

fillRect()와는 다르게 fillStyle로 꽉채운 방식이 아닌 선으로 직사각형을 그린다.

CanvasRenderingContext2D.beginPath()

본격적으로 그리기에 앞서서 새로운 경로를 시작한다.

CanvasRenderingContext2D.closePath()

현재의 지점에서 현재 부수 경로의 시작까지의 직선을 추가한다. 만약에 도형이 이미 닫혀졌다면 closePath 메서드는 아무것도 하지 않는다.

CanvasRenderingContext2D.arc()

현재의 부수 경로에 원형 아치를 그린다.

arc 함수는 (x, y)좌표를 기준으로 radius로 아치를 그린다. 경로는 startAngle에서 시작하고 endAngle에서 끝난다.

CanvasRenderingContext2D.bezierCurveTo()

배지어 곡선을 그린다. 3가지 지점을 요구하면 첫번째와 두번째 점들은 컨트롤 지점이고 마지막 하나는 종점이다.

What is URL?

URL(Uniform Resource Locator)
웹에 퍼블리시된 리소스를 반환하기 위해 브라우저에 의해 사용되는 구조(메커니즘)이다.

URL은 웹상에서의 리소스의 고유한 주소 그이상도 그이하도 아니다. 이러한 리소스들은 HTML 페이지, CSS 문서, 이미지… 일 수가 있다.
URL의 예외 사항에는 URL이 가리키는 리소스가 더이상 존재하지 않거나 이동될 경우이다.

리소스들은 URL로 대표되며, URL 그자체는 웹 서버에 의해 다루어진다. 따라서 리소스와 이에 연관된 URL을 다루는 건 웹 서버 주인의 몫이다.

URL의 구조

URL은 scheme, Domain Name, Port, Path to the file, Parameter, Anchor로 이루어져있다

Scheme

브라우저가 리소스를 불러오기 위해 사용해야할 protocol(통신 규약)을 의미하며 이중에는 주로 HTTP와 HTTPS가 쓰인다.

  • protocold은 컴퓨터 네트워크에서의 데이터를 교환하거나 전달할때 사용하는 method들이다.)

Authority

Authority는 Domain Name과 Port로 나누어져 있다.
Port는 web server로 접근하기 위한 기술적인 관문을 의미한다.

Path to resource

Web server위에서 resource로의 길을 의미한다.

Parameter

?key1=value1&key2=value2는 web server에 제공된 추가 parameter이다. Parameter들은 & symbol로 나누어진 key/value 쌍들을 가진 list이다.
웹 서버는 이러한 parameter들을 리소스를 실행하기 전에 추가적 기능을 할 때 사용할 수 있다. 각각의 웹서버들은 parameter에 따라 고유한 규칙들이 있으며 웹서버가 parameter들을 관리하는지 알기위해서는 web server 주인을 물어보는게 옳다.

Asynchronous(비동기화)

함수를 호출하면 함수 코드가 평가되어 함수 실행 컨텍스트가 실행된다. 이때 생성된 함수 실행 컨텍스트는 실행 컨텍스트 스택(콜 스택이라고도 부른다)에 푸시되고 함수 코드가 실행된다. 함수 코드의 실행이 종료하면 함수 실행 컨텍스트는 실행 컨텍스트에서 팝되어 제거된다.

실행 중인 태스크가 종료할 때까지 다음에 실행될 태스크가 대기하는 방식을 동기(synchronous)처리라고 한다. 동기 처리 방식은 태스크를 순서대로 하나씩 처리하므로 실행 순서가 보장된다는 장점이 있지만, 앞선 태스크가 종료할때까지 이후 태스크들이 blocking 되는 단점이 있다.

반대로 현재 실행 중인 태스크가 종료되지 않은 상태라 해도 다음 태스크를 곧바로 실행하는 방식을 비동기(asynchronous)처리라고 한다. => blocking은 안하지만 task의 실행 순서가 보장되지 않는 단점이 있다.
비동기 처리를 위한 콜백 패턴은 call back hell을 발생시켜서 가독성을 나쁘게 하고, 비동기 처리 중 발생한 에러의 예외 처리가 곤란하며, 여러 개의 비동기 처리를 한번에 처리하는 데도 한계가 있다.

비동기 함수는 task que에 push되어 대기하다가 call stack이 비게 되면, 다시 말해서 전역 코드 및 명시적으로 호출된 함수가 종료되면 비로소 call stack에 push되어 실행된다.

JavaScript는 single thread 방식으로 동적한다. Single thread로 동작하는 것은 browser가 아니라 browser에 내장된 JavaScript Engine이라는 것에 주의하기 바란다. 만약 모든 JavaScript코드가 Javascript Engine에서 single thread 방식으로 동작한다면 비동기적 방법으로 동작할 수 없다.

Timer

함수를 일정 시간이 경과된 이후에 호출되도록 함수 호출을 예약하려면 타이머 함수를 이용하며 이를 호출 스케줄링(Scheduling a call)이라고 한다.

자바스크립트 엔진은 단 하나의 실행 컨텍스트 스택을 갖기 때문에 두가지 이상의 태스크를 동시에 실행할 수 없다. 즉 자바스크립트 엔진은 Single Thread 방식으로 동작한다. setTimeout 과 setInterval은 비동기(asynchronous)방식으로 동작한다.

setTimeout/ clearTimeout

setTimeout 함수는 두번째 인수로 전달받은 시간으로 단 한 번 동작하는 타이머를 생성한다.
이후 타이머가 만료되면 첫 번째 인수로 전달받은 콜백 함수가 호출된다. 즉, setTimeout 함수의 콜백함수는 두 번째 인수로 전달받은 시간 이 후 단 한 번 실행되도록 호출 스케줄링된다.

setInterval/clearInterval

setInterval 함수는 두번째 인수로 전달받은 시간으로 반복 동작하는 타이머를 생성한다. 타이머가 만료될때 마다 첫번째 인수로 전달받은 콜백 함수가 반복 호출된다. 이건 타이머가 취소될 때까지 계속된다.

Debounce

짧은 시간 간격으로 이벤트가 연속적으로 발생하면 이벤트 핸들러를 호출하지 않다가 일정 시간이 경과된 이후에 이벤트 핸들러가 한 번만 호출되도록 한다.
즉 짧은 시간 간격으로 발생하는 이벤트를 그룹화해서 마지막에 한 번만 이벤트 핸들러가 호출되도록 한다.

Input 이벤트는 사용자가 텍스트 입력필드에 값을 입력할 때마다 연속해서 발생한다. 서버에 부담을 주지않기 위해 입력을 완료했을때만 요청을 전송하는 것이 바람직하다.

하지만 사용자가 입력을 완료했는지는 알기가 어렵기때문에 일정 시간동안 텍스트 입력필드에 값을 입력하지 않으면 입력이 완료된 것으로 간주한다.

Throttle

짧은 시간 간격으로 이벤트가 연속해서 발생하더라도 일정 시간 간격으로 이벤트 헨들러가 최대 한번 호출되도록 한다.
